## Terminology

The cryptography and encryption topic seems hard first of all because tons of abbreviations and terms that seem unfamiliar. One of the first steps towards understanding is being able to easily operate by these terms.

For now you can skip reading them, and just return to this chapter as you read the article.

**TLS** - Transport Layer Security

**RSA** (**Rivest–Shamir–Adleman**) - both encryption and signature algorithm, pretty heavy and now considered legacy in TLS.

**MAC** (**Message Authentication Codes aka Authentication tag**) - sort of signature, that is computed over the payload and added to it to confirm that the payload hasn’t been changed. Typically it is implemented as some hashing with MAC secret key involved. Examples: **GHASH**, **HMAC**.

**AEAD** (**Authenticated Encryption with Associated Data**) -  encryption + signature (to ensure encrypted data was not tempered). The most commonly used AEAD is **AES-GCM**, which computes encrypted bytes (AES part)  in parallel per block and then generates **GHASH MAC **over them (GCM part)

**HKDF **(**HMAC-based Key Derivation Function**) - function that can deterministically derive a bunch of different keys needed for signing and encrypting, from a single shared key.

**ECDSA **(**Elliptic Curve Digital Signature Algorithm**) - signature creation and verification approach using public and private key derived from Elliptic Curve formula

**ECDH **(**Elliptic Curve Diffie-Hellman**) / **ECDHE **(**Elliptic Curve Diffie-Hellman Ephemeral**) - shared secret derivation from private and public keys using Elliptic Curve formula.

**ASN.1 (Abstract Syntax Notation One)** - schema/structure definition, like JSON.

**DER (Distinguished Encoding Rules) **- binary encoding for **ASN.1** data

**PEM **- textual (base64) encoding of **DER**, with headers marking is the payload {-----BEGIN PRIVATE KEY-----}, {-----BEGIN CERTIFICATE-----}, {-----BEGIN PUBLIC KEY-----}

**TbsCertificate** - To Be Signed Certificate, contains subject, issuer, validity, **public key**, serial number, extensions.

**Certificate **- is a specific **ASN.1** structure (**Certificate ::= SEQUENCE { tbsCertificate, signatureAlgorithm, signatureValue }** ) encoded as **DER **(optionally encoded to **PEM**).


## What is TLS and why bother?

TLS - is a protocol that is needed to ensure the communication over the network is secured, not readable and not malformed by a middleman.

Keep in mind we won’t consider SSL here as it is deprecated and not used pretty much.

Having in mind our TCP / IP stack, where is it? Let’s recall the stack itself [TCP/IP Illustrated Volume 1 by Richard Stevens]:


![alt_text](images/image1.png "image_tooltip")


TLS / SSL is the layer that is working on top of the Transport layer (TCP as delivery guarantee is required). Worth noting, that default TLS/SSL implementation is run in user space rather than kernel space.

 \
But still it is hard to put it into the Application layer fully, as TLS will be an intermediate layer between Transport (TCP) and Application (HTTP). Under the hood SSL is going to be just a user space library running or used by Application server, so for TCP it will be opaque, as TCP just operates over the bytes (does not matter encrypted or not).

So at the same time TLS is considered as a separate layer, as part of Application layer and as NOT a layer at all.


![alt_text](images/image2.png "image_tooltip")



### The purpose

The application of the TLS is very simple, when you transfer data over the network - typically you don’t want it to be readable by other parties, and you don’t want to receive malformed data without knowing it.

These are the problems that are solved by **TLS**:



* Confidentiality (Encryption): it encrypts the data transferred between client and server, so whoever captures the packet won’t be able to decrypt it
* Authentication (Signature): it ensures data is coming from trusted side (you don’t want mistakenly send card number to malicious actor who tricked you to believe this is the bank e.g. via Typosquatting)
* Integrity (Signature): it ensures the data wasn’t malformed while transferred over the network, so servers receive exactly what the clients have sent.


### Encryption

Encryption is used to transform original text to bytes stream that is completely opaque when intercepted, that therefore is untransformed on the receiving side using some sort of keys.

We have 2 different types of encryption: asymmetric and symmetric, they have different pros/cons and serve different purposes usually.


#### Asymmetric encryption

Needs 2 keys for encryption: public and private.

It all works because of math magic with prime numbers. In simple words you can find two numbers `private_key` and `public_key` such that: 

`encrypt(message, public_key) == decrypt(message, private_key)`.


![alt_text](images/image3.png "image_tooltip")


For client/server communication we do:



* Server generates 2 keys: `sprivate`, `spublic`
* Client generates 2keys: `cprivate`, `cpulbic`
* Server provider `spublic` to client, client provider `cpublic` to server.
* Server sends a message and encrypts it with `cpublic`. Client decrypts with `cprivate`
* Client sends a message and decrypts it with `spublic`. Server decrypts with `sprivate`

The math magic is called **RSA** (Rivest–Shamir–Adleman, most popular asymmetric encryption and signature algo), that works via generating 2 huge prime numbers, deriving private and public key from them, then encrypting/decrypting using these private and public keys. 

Note still that these keys are essentially big numbers (bytes) as well as the cipher and message. The trick here is knowing only the public key - it is very hard/almost impossible to bruteforce private key -> therefore decrypt.

We see the main benefit of asymmetric encryption: we don’t have problems sharing the key, the public key is very safe to distribute, but on the other hand - it is extremely slow for encryption, needs more CPU and time compared to symmetric encryption.


#### Symmetric encryption

Works over a single shared key that both server and client know. Shortly speaking, encryption is usually deterministic bytes operations performed with shared secret + bunch of derived keys thus making it reversible. \
 \
For client/server communication we do:



* Somehow distribute the same shared key to client and to server
* Both server and client derive 2 keys: **server->client shared key** used only when server sends message to client , and **client->server shared key** used only when client sends message to server. Both of them generate a bunch of other secret keys that are used for encryption and signature. It is done so to make it less guessable.
* Client encrypts data and signs with client -> server shared key. Server decrypts it using the same **client -> server shared key**.
* Server encrypts data and signs with client -> server shared key. Client decrypts it using the same **client -> server shared key**.


![alt_text](images/image4.png "image_tooltip")


Symmetric encryption is faster than asymmetric. There is only one problem: how to exchange the shared key securely?


## Signature

Signature is bytes computed from some payload using some cryptographic algorithms (RSA, or ECDSA), that can be used to prove that message wasn’t malformed and was sent from the real owner.

It does not change the payload itself (it stays plain and visible to any middleman), but it adds an additional message/part in the end, that the receiver can use to validate the integrity and authenticity of the payload.

Signatures are mainly used in TLS certificates to validate the certificate itself, including the chain of parent certificates.

There are 2 main algorithms: **ECDSA **(ES256) and **RSA** (RS256).

ECDSA is mainly used as it is easier to compute (requires less CPU and time), it needs less space for signature still providing the same security guarantees.


### ECDSA (Elliptic curve)

In modern TLS the most useful algorithm for signatures is ECDSA, which is a geometry based method based on elliptic curve function. 


![alt_text](images/image5.png "image_tooltip")


To not put too much math details:



* signing: we are getting the starting point on the elliptic curve, and performing scalar multiplication `private_key` number of times (point addition) thus getting the end value (`public_key`, it is just X coordinate). Then we compute the `result` using formula from `private_key`, hash of the payload and `public_key`. 
* verifying: receiver is given `result`, `public_key` and payload. By using reversed formula it is possible to verify that computed x coordinate (`public_key`) and passed x coordinate do match


![alt_text](images/image6.png "image_tooltip")



### RSA

Signing using RSA is pretty similar to encryption. The same math trick is used: you are getting two huge prime numbers, and derive private and public keys from them. The signature is just an encrypted cipher over the hashed payload (with different padding due to security).


![alt_text](images/image7.png "image_tooltip")



## Agree on shared key / ECDHE (Elliptic curve)

Previously we concluded that symmetric encryption is cheaper and faster than asymmetric.

There is only one concern left: how can we agree on the same shared secret over an untrusted network?

Mathematicians have solved this problem for us engineers a long time ago. The method is called **ECDHE **(Elliptic Curve Diffie-Hellman Ephemeral), yes, again elliptic curves.


![alt_text](images/image8.png "image_tooltip")


It works using the same private/public keys derivation using elliptic curve:



* Client is getting the starting point on the elliptic curve, and performing scalar multiplication `private_key` number of times (point addition) thus getting the end value (`public_key`, it is just X coordinate).
* The server is getting the starting point on the elliptic curve, and performing scalar multiplication `private_key` number of times (point addition) thus getting the end value (`public_key`, it is just X coordinate).
* Client gets Server’s public_key
* Server gets Client’s public_key
* Both client and server will arrive into the same `shared_secret` given the keys they have, because of the math properties of elliptic curves.

Note that this `shared_key` is never used raw, we are running HKDF functions to derive more keys from this single shared key to start using them for symmetric encryption.


## Certificates

This was probably one of the most mysterious topics for me. Terminology helped me.

Certificate is just a bunch of metadata: issuer, subject name, expiry, etc, along with a public key. All this metadata is signed using some private key. Now 3 parts: metadata, public key and signature are presented in a special format called **ASN.1**.

**DER** - is binary encoding for **ASN.1** data. **PEM**, and .pem files are just textual (base64) encoding for **DER**.

Certificate generated using usually ECDSA pair, where public key is encoded into certificate and signature is produced using private key and body of certificate.

You can think about the certificate as a JWT that the server presents to you, so that you can verify that this website is legitimate.


![alt_text](images/image9.png "image_tooltip")


Generally everyone can just generate an ECD key pair and issue a self-signed certificate, and then replace it as a middleman on the way to the client. That way, the client will agree on a shared key with the middleman, and the middleman can serve as a proxy decrypting and reading all the messages.

How does the client then differentiate which certificate was really issued by server and which one was not?


### PKI / Chain of Trust

People generally agreed on few Root CA (certificate authorities) that can issue new certificates. Root CA is just a server that holds private key and public certificate, and can issue new certificates using its private key.


![alt_text](images/image10.png "image_tooltip")


Since the internet grew, and private keys are extremely sensitive information, we have added a new layer of certificates called Intermediate CA. These are usually used to issue new certificates.

All browsers agreed to distribute Root CA public keys so that clients can easily tell whether leaf + intermediate certificates are coming from root or not.


![alt_text](images/image11.png "image_tooltip")


During TLS handshake the certificate chain is validated like this:



* server signs all previous handshake tls messages with Leaf server’s private key
* client then: 
    * validates that signature with leaf certificate that server presents
    * client validates leaf certificate with intermediate certificate (that server attaches too)
    * client validates intermediate certificate against root certificate that is already distributed to the client


![alt_text](images/image12.png "image_tooltip")



## How does TLS work?


![alt_text](images/image13.png "image_tooltip")


Now given all prerequisite background we can grasp what is going on in TLS layer:



* **ClientHello**: client generates ECDHE keys `c_priv` and `c_pub` and send `c_pub` along with supported ciphers, curves, nonce, etc.
* **ServerHello**: server generates ECDHE keys `s_priv` and `s_pub`, sends chosen cipher, nonce, etc.
* Client and Server compute `shared_	secret` using ECDHE algorithm. `Server’s shared = ECDHE(s_priv, c_pub)`, `Client’s shared = ECDHE(c_priv,j s_pub)`
* From `shared_secret` they derive a bunch of keys that are going to be used for actual encryption using the HKDF function.
* **Certificate**: Server sends certificate to the client
* **CertificateVerify**: Server sends hash over all the previous TLS messages along with signature created from Leaf’s private key)
* **ServerFinished**: Server computes the final e2e encryption key along with HMAC over all the messages sent to the client so far.
* Client validates certificate chain (leaf -> intermediate, root)
* Client validates the leaf certificate from CertificateVerify message and Leaf cert’s public key
* **ClientFinished**: Client verifies HMAC from Finished message matches the one it calculated from all the message it received, and sends its HMAC to the server.
* Server validates client’s Finish HMAC.

The beauty of this method is everything is happening over an open network, and any middleman can sit there and see the communication, but couldn’t do anything about it. Besides that the encryption keys generated as well as shared_secret are only for the current TLS session. Even if they are compromised - the malicious actor can decrypt a single session message.


## Practical application

One of the most common applications of all above that I have seen is mTLS over the internal components.

mTLS - is mutual TLS, meaning server1 presents its own certificate to server2 as well as server2 presents its own certificate to server1. Thus they both authorize each other, so not only server1 knows it is talking to trusted server2, but also server2 knows the request is coming from the trusted server1.

Another approach that I have seen: custom PKI, when you control how, where and to whom the certificates can be issued. You could build different chains of trust thus limiting the disjoint set of components that can talk to each other.

The last example would be the case, when you want to establish ephemeral encryption key per session without TLS: you can implement custom ECDHE (using libraries ofc).
